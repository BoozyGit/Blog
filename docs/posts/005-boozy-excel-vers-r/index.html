Le constat de Boozy

Dans de nombreux projets d’ONG ou de thèses en Afrique francophone, Excel est la première — et souvent la seule — porte d’entrée vers les données. C’est compréhensible : c’est visuel, familier, et ne demande aucune installation particulière.

Mais quand le volume augmente, quand il faut refaire la même analyse sur dix fichiers différents, quand le directeur de thèse demande des modifications à 23h la veille de la soutenance… Excel devient un piège.

🔀 Technique × Contexte ONG (post #21)

Imaginez Amara, coordinatrice de projet dans une ONG de santé à Bamako. Elle reçoit chaque mois les données de 12 centres de santé dans 12 fichiers Excel différents. Sa mission : consolider tout ça en un rapport mensuel pour les bailleurs. Chaque mois, elle passe deux jours entiers à copier-coller, reformater, corriger. Deux jours sur vingt jours ouvrables. C'est 10% de son temps de travail — évaporé.

📊 Excel — Le piège familier

❌Manuel et fragile — Un clic de travers, une cellule écrasée, et vos résultats sont silencieusement faussés. Sans alerte.
❌Non reproductible — "Comment as-tu obtenu ce résultat ?" — "Euh… une série de clics que je ne me rappelle plus."
❌Formules illisibles — =SI(NB.SI(A2:A100;"")>0;INDEX(...)) n'est pas de la science, c'est de l'archéologie.
❌Ne scale pas — 500 lignes, ça va. 50 000 lignes ? Excel rame, plante, ou produit des résultats silencieusement erronés.
❌Zéro statistique avancée — Régression logistique, survie, méta-analyse ? Il faudra des add-ins bricolés ou changer d'outil de toute façon.

📦 R — La liberté du script

✅100% traçable — Chaque transformation est écrite noir sur blanc. Auditez, partagez, justifiez.
✅Reproductible — Données mises à jour ? Relancez le script : nouveaux résultats en secondes.
✅Lisible avec Tidyverse — data |> filter(age > 18) |> group_by(sexe) — c'est presque de l'anglais.
✅Scale sans limite — 500 ou 5 millions de lignes, R ne voit pas la différence.
✅Tout en un — Nettoyage, stats, visualisation, rapport, carte, méta-analyse. Un seul outil pour tout.

Le décryptage : la puissance du “Script”

La rupture fondamentale entre Excel et R tient à un seul concept : la reproductibilité computationnelle.

En Excel, votre analyse est une série de gestes. Quand vous fermez le fichier, ces gestes disparaissent. Il ne reste que le résultat — impossible à auditer, impossible à vérifier, impossible à corriger proprement si une erreur est découverte six mois plus tard.

En R, votre analyse est un texte — un script. Ce texte est :

  • Lisible par un humain (vous dans 6 mois, votre directeur, un reviewer)
  • Exécutable par une machine (résultats reproductibles à l’identique)
  • Versionnable avec Git (on y reviendra dans le post #28)

💡 Ce que dit la science

Une étude publiée dans PLOS ONE (Stodden et al., 2018) a testé la reproductibilité de 204 articles de sciences computationnelles : moins de 26% des analyses pouvaient être reproduites avec les données et le code fournis. La crise de reproductibilité n'est pas qu'une question de fraude — c'est souvent une question d'outillage. Excel en est l'un des facteurs.

Le tableau de correspondance Excel → R

Voici les équivalences directes pour les tâches les plus courantes. Rien d’intimidant — chaque ligne R est plus courte que la formule Excel correspondante.

Ce que vous faites dans Excel L'équivalent en R (Tidyverse) Gain
Filtrer des lignes (AutoFiltre) filter(age > 18, sexe == "F") ✅ Reproductible
Sélectionner des colonnes select(id, age, diagnostic) ✅ Traçable
Créer une nouvelle colonne mutate(imc = poids / taille^2) ✅ Versionnable
Tableau croisé dynamique group_by(groupe) |> summarise(n()) ✅ Automatisable
Trier les données arrange(desc(date_inclusion)) ✅ Scriptable

Remplacer les valeurs manquantes replace_na(list(poids = 0)) ✅ Documenté

L’astuce de Boozy pour se lancer sans paniquer

Boozy a failli rater son décollage avec R à cause d’une erreur classique : vouloir tout apprendre avant de commencer. Voici les trois règles qui l’ont sauvé.

  • 1

    Ne convertissez pas tout — commencez par une seule tâche

    Choisissez la manipulation Excel qui vous fait le plus perdre de temps. Juste celle-là. Cherchez comment la faire en R. Maîtrisez-la. Puis passez à la suivante. L'erreur classique : vouloir "migrer vers R" en un week-end.

  • 2

    Installez le Tidyverse — votre porte d'entrée non-informaticien

    Le Tidyverse est une collection de packages R conçue pour que le code ressemble à du langage naturel. filter(), select(), mutate(), summarise() — des verbes anglais simples. C'est délibéré : ses créateurs voulaient que des chercheurs en sciences sociales, biologie et santé publique puissent l'utiliser sans formation informatique.

  • 3

    La méthode "Copier-Adapter-Avancer"

    Pas besoin d'écrire du code parfait. Trouvez un exemple qui ressemble à ce que vous voulez faire (Stack Overflow, QIBAR Blog, documentation Tidyverse), copiez-le, adaptez les noms de colonnes à vos données, lancez-le. C'est comme ça que 90% des data scientists apprennent encore aujourd'hui.

Le script de Boozy — son tout premier nettoyage de données

Voici exactement ce que Boozy a fait lors de sa première semaine avec R. Un script de nettoyage simple, commenté ligne par ligne, pour nettoyer un fichier de données collecté sur le terrain.

# ── Installer les packages (une seule fois) ──────────────────────────────
# install.packages("tidyverse")
# install.packages("readxl")

# ── Charger les librairies ───────────────────────────────────────────────
library(tidyverse)   # pour tout : filter, mutate, select, etc.
library(readxl)      # pour lire les fichiers Excel

# ── 1. Importer les données ──────────────────────────────────────────────
# On lit le fichier Excel directement — fini le copier-coller
donnees_brutes <- read_excel("donnees_terrain.xlsx", sheet = "Feuil1")

# Aperçu rapide : dimensions et premières lignes
glimpse(donnees_brutes)

# ── 2. Renommer les colonnes (noms propres, sans espaces) ────────────────
donnees <- donnees_brutes |>
  rename(
    id          = `N° Patient`,
    age         = `Age (ans)`,
    sexe        = `Sexe`,
    diagnostic  = `Diagnostic principal`,
    date_visite = `Date de visite`
  )

# ── 3. Nettoyer les types de variables ───────────────────────────────────
donnees <- donnees |>
  mutate(
    # Convertir la date (souvent mal lue depuis Excel)
    date_visite = as.Date(date_visite),
    
    # Standardiser la variable sexe (majuscules, espaces)
    sexe = str_to_upper(str_trim(sexe)),
    sexe = case_when(
      sexe %in% c("M", "MASCULIN", "HOMME") ~ "Masculin",
      sexe %in% c("F", "FEMININ", "FEMME")  ~ "Féminin",
      TRUE ~ NA_character_   # tout le reste → valeur manquante
    ),
    
    # Supprimer les âges impossibles (< 0 ou > 120)
    age = if_else(age >= 0 & age <= 120, age, NA_real_)
  )

# ── 4. Gérer les doublons ─────────────────────────────────────────────────
n_avant <- nrow(donnees)
donnees <- donnees |> distinct(id, .keep_all = TRUE)
n_apres <- nrow(donnees)
cat("Doublons supprimés :", n_avant - n_apres, "\n")

# ── 5. Rapport de qualité des données ─────────────────────────────────────
# Pourcentage de valeurs manquantes par colonne
donnees |>
  summarise(across(everything(), ~ mean(is.na(.)) * 100)) |>
  pivot_longer(everything(), names_to = "variable", values_to = "pct_manquant") |>
  arrange(desc(pct_manquant))

⏱️ Comparaison de temps — cas réel

Excel : 4h pour Amara (coordinatrice ONG) chaque mois — copier-coller, reformater, vérifier manuellement.
R : 45 minutes pour écrire le script une fois. Ensuite : 30 secondes par mois pour le relancer sur les nouveaux fichiers.

Sur 12 mois : Excel = 48h perdues. R = 45min d'investissement + 6min de travail. L'équivalent de 6 jours de travail récupérés.
